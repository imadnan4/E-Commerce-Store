Phase 1 — Project setup

Before writing a single line of app code, get your environment, database, and project skeleton ready.

Virtual environment

Isolates your project's Python packages from the rest of your system.

PostgreSQL

Create a database manually in psql before Django can connect to it.

.env file

Keep secrets like DB passwords out of your code using environment variables.

1
Create a virtual environment & install packages

Run these commands inside your project folder:

  • python -m venv venv — creates the isolated environment
  • source venv/bin/activate — activates it on Linux
  • pip install django psycopg2-binary python-dotenv Pillow

Understand what each package does before installing it. psycopg2-binary is the PostgreSQL driver. Pillow handles product image uploads.

2
Create the PostgreSQL database

Open psql and create your database before touching Django settings:

  • psql -U postgres
  • CREATE DATABASE ecommerce_db;
  • \q to exit

Django doesn't create the database for you — it only creates tables inside an existing one.

3
Start a Django project and two apps

Django separates a "project" (config) from "apps" (features):

  • django-admin startproject ecommerce .
  • python manage.py startapp users
  • python manage.py startapp store

The dot in the first command puts settings in the current folder instead of a nested one. Register both apps in INSTALLED_APPS inside settings.py.

Milestone: Run python manage.py runserver and see the Django rocket page at localhost:8000. If you do, setup is complete.

Phase 2 — How Django works (MVT)

Django follows a Model–View–Template pattern. Understand this flow before writing any feature.

URL → View → Template

Every browser request hits a URL, which calls a view function, which renders a template.

Context dictionary

Views pass data to templates via a Python dict. Template variables come from this dict.

URL namespacing

include() connects app-level urls.py files to the project-level one.

1
Wire up your first URL → view → template
Django

Before building anything, manually trace a request from browser to screen:

  • Add a path to store/urls.py pointing to a home view
  • Include store urls in ecommerce/urls.py with include('store.urls')
  • Write a home(request) function in store/views.py that returns render(request, 'store/home.html', {})
  • Create the template file and put a simple <h1>Hello</h1> in it
2
Understand Django templates and template inheritance
Django

All your pages will extend base.html. Learn these template tags:

  • {% extends 'base.html' %} — inherits from a parent
  • {% block content %}{% endblock %} — defines a replaceable slot
  • {% include 'navbar.html' %} — inserts a partial file
  • {{ variable }} — outputs a context value
  • {% for item in list %}{% endfor %} — loops
3
Configure static files and media files
Django

Static files (CSS, JS) and media files (uploaded images) are handled differently in Django:

  • Add STATICFILES_DIRS and MEDIA_ROOT in settings
  • Use {% load static %} and {% static 'css/style.css' %} in templates
  • Serve media during development by adding + static(MEDIA_URL, ...) in main urls.py
Milestone: You should be able to visit a URL, have it call a view, and see a styled HTML page with your CSS loaded.

Phase 3 — Models & the database

Models are Python classes that become PostgreSQL tables. This is the heart of your app's data.

Model = table

Each class attribute becomes a column. Django's ORM translates Python to SQL for you.

ForeignKey

Links two models (tables) together. An order belongs to a user via a ForeignKey.

Migrations

makemigrations generates SQL. migrate runs it. Run both after every model change.

1
Write the Category and Product models
Database

Think about what data each product needs before coding:

  • Name, description, price, stock count, image, availability flag
  • A SlugField for clean URLs like /product/blue-sneakers/
  • A ForeignKey to Category so products can be filtered

After writing models, always run makemigrations store then migrate.

2
Write the Order and OrderItem models
Database

An order contains many products. Model this as two tables:

  • Order — belongs to a user, has status, shipping address, total
  • OrderItem — one row per product in an order, with quantity and price snapshot
  • Why snapshot the price? Product prices can change after purchase
3
Register models in admin and add test data
Database

Django's admin panel lets you add data without writing SQL:

  • Register models with @admin.register(Product) in admin.py
  • Use prepopulated_fields = {'slug': ('name',)} so slug auto-fills
  • Create a superuser with python manage.py createsuperuser
  • Add 3-4 categories and 6-8 products via /admin
Milestone: You can open /admin, add a product with an image, and query it in a view using Product.objects.all().

Phase 4 — User authentication

Django has a built-in auth system. Use it rather than building one from scratch.

Built-in User model

Django provides a User model with username, password, email — already battle-tested.

@login_required

A decorator that blocks non-logged-in users from accessing a view and redirects to login.

Profile model

Extends User via OneToOneField to store extra info like address and phone number.

1
Build the registration view with UserCreationForm
Auth

Django's UserCreationForm gives you a ready-made form with validation:

  • Extend it to add email, first name, last name fields
  • In the view: if POST and form valid → form.save() → login(request, user) → redirect
  • On GET: just render the empty form
2
Build the login and logout views
Auth

Use Django's built-in AuthenticationForm and login()/logout() functions:

  • authenticate() checks credentials and returns a user or None
  • login(request, user) creates the session
  • logout(request) clears it
  • Handle the ?next= redirect parameter so users return to where they were
3
Create a Profile model linked to User
Auth

The built-in User doesn't have fields for phone or address. Extend it:

  • Create a Profile model with OneToOneField(User, on_delete=models.CASCADE)
  • Create a Profile when a user registers: Profile.objects.create(user=user)
  • In profile view, use get_or_create as a safety net
Milestone: A user can register, log in, see their name in the navbar, visit their profile, and log out. The @login_required decorator redirects anonymous users to the login page.

Phase 5 — Store logic (cart & orders)

The session-based cart is the most important concept here. Orders persist to the database.

Session-based cart

Cart data lives in request.session — a server-side dictionary tied to the browser session, no login required.

Order creation

On checkout, read the session cart, create an Order row, then create one OrderItem per product.

Context processor

Injects data like cart count into every template automatically — no need to pass it in every view.

1
Implement the session cart (add, remove, update)
Django

The cart is stored as a dict in the session: {'product_id': {'quantity': 2, 'price': '29.99'}}

  • Read cart: cart = request.session.get('cart', {})
  • Add item: check if key exists, increment or set to 1
  • Write back: request.session['cart'] = cart — Django won't save unless you reassign
  • Keys must be strings since JSON serializes dict keys as strings
2
Build the cart view and cart template
Django

The cart view reads the session, fetches each Product from DB, and builds a list to pass to the template:

  • Loop over session cart items, fetch each Product by ID
  • Calculate item_total = price * quantity in Python, not the template
  • Handle Product.DoesNotExist in case a product was deleted
3
Build the checkout and order creation flow
Database

Checkout converts the session cart into real database rows:

  • Require @login_required — orders need a user
  • On POST: create one Order object, then loop cart and create OrderItem per product
  • Decrement product.stock per item sold
  • Clear the cart: request.session['cart'] = {}
  • Redirect to order success page with the order ID
4
Write the cart_count context processor
Django

A context processor is a function that runs on every request and adds to the template context:

  • Create store/context_processors.py with a function that returns {'cart_count': N}
  • Register it in settings.py under TEMPLATES → OPTIONS → context_processors
  • Now {{ cart_count }} works in every template including the navbar
Milestone: You can add products to the cart, see the count in the navbar update, visit the cart page, and complete a checkout that creates a real Order row visible in the admin panel.

Phase 6 — Frontend (templates & CSS)

Plain HTML + CSS with Django template tags. No framework needed for this project.

CSS Grid

Use grid-template-columns: repeat(auto-fill, minmax(240px, 1fr)) for a responsive product grid.

Django forms in templates

Loop over form fields with {% for field in form %} to render inputs with labels and errors.

CSRF token

Every POST form needs {% csrf_token %} inside it. Django rejects forms without it.

1
Build base.html with template blocks
Frontend

All pages extend this. Define at minimum:

  • {% block title %}{% endblock %} in the <title> tag
  • {% block content %}{% endblock %} in the <main> tag
  • Include the navbar partial with {% include 'navbar.html' %}
  • Show messages (flash notifications) between navbar and content block
2
Style the product grid and cards
Frontend

The product grid is the most visible part of the store:

  • Use CSS Grid with auto-fill so cards wrap on small screens automatically
  • Give cards a hover effect with transform: translateY(-4px) and a box-shadow change
  • Use object-fit: cover on product images so they all have the same height
3
Handle category filtering in the home view
Django

Filtering is done via a query parameter like /?category=electronics:

  • Read with request.GET.get('category') in the home view
  • If present, filter products: Product.objects.filter(category__slug=slug)
  • Pass selected_category to template to highlight the active filter button
Milestone: The store looks clean, category buttons filter products, product cards link to detail pages, and the whole UI is responsive on mobile.
